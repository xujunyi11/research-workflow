# 科研文献支线工作流（仅 Chat）

本支线只用 Chat 完成论文理解和跨论文判断，不调用 Work。上传、下载、命名和文件替换可由用户手动完成，也可由 Codex 作为操作与文档管理员协助；Codex 不分析论文、不补写科研结论。

## 一次性部署

1. 新建独立 ChatGPT 项目，建议命名“科研文献知识库”。
2. 把 `Chat项目指令.md` 全文粘贴到项目指令。
3. 将 `知识库文件模板.md`、`主线交接模板.md`、`知识库/领域概览.md` 和 `知识库/论文目录.md` 添加为项目来源。前两份是稳定格式规范，后两份是持续更新的知识文件。
4. 不把全部论文、论文卡和主题知识长期加入项目来源；每次只在消息中附带当前论文和直接相关文件。
5. 在本地建立 `知识库/论文卡/`、`知识库/主题知识/`、`文献问题/`、`文献地图/`。Chat 输出完整文件后，由你保存到对应位置。

项目来源需要更新时，先删除旧版同名文件，再上传新版，避免同名来源并存。历史聊天只提供讨论上下文，不作为证据。

## 两个模板如何使用

模板是工作流规范，不是每批生成的产物，也不是科研证据。完成一次部署后无需每批重复上传；只有模板内容发生修改时，才在项目来源中删除旧版并上传新版。

### `知识库文件模板.md`

- 只配置在文献支线项目来源，不交付给主线作为知识来源。
- 规定论文卡、论文目录、领域概览、主题知识的文件头、字段和正文结构。
- 模式 A 中，Chat 按该模板为本批每篇论文分别生成论文卡，并输出完整新版论文目录；需要更新领域概览或主题知识时，也必须沿用其中对应结构。
- 用户或 Codex 落盘前检查 `document_type`、`revision`、`based_on`、`replace_target` 和 `status`；缺字段时要求 Chat 修复，不能由 Codex补写科研内容。

### `主线交接模板.md`

- 同时配置在主线项目和文献支线项目来源，作为双方共享的交接契约。
- 主线模式在需要文献核查时使用“文献问题”部分，生成 `文献问题_vNNN.md` 并交给支线。
- 支线模式 B 使用“文献地图”部分回答，生成同编号 `文献地图_vNNN.md` 并交回主线。
- `vNNN` 每次新问题递增，问题与地图编号必须相同；交接文件永不覆盖。模板文件本身不随问题递增版本。

最小使用链路：主线按模板生成文献问题 → 支线上传该问题及相关知识文件 → 支线按模板返回同编号文献地图 → 主线据此更新或保留当前 IDEA。

## 支线产物与主线交付

| 支线产物 | 用途 | 是否交付主线 | 交付方式 |
|---|---|---|---|
| `知识库/领域概览.md` | 跨主题问题结构、研究演进与高层共识 | 是 | 最新版作为主线项目常驻来源 |
| `知识库/论文目录.md` | 覆盖范围、主题检索和论文卡定位 | 是 | 每批处理完成后，把最新版同步为主线项目常驻来源 |
| `知识库/主题知识/*.md` | 具体主题的证据综合、方法关系、争议和研究缺口 | 是 | 只交付已经完成综合的最新版，并作为主线项目常驻来源 |
| `知识库/论文卡/*.md` | 单篇论文的完整证据与架构记录 | 按需 | 主线需要核验具体论文时临时附带，不设为常驻来源 |
| `文献地图/文献地图_vNNN.md` | 回答主线提出的一个明确文献问题 | 是 | 返回对应主线对话，作为一次性决策材料 |
| 原始论文 PDF | 原始证据 | 按需 | 仅在论文卡不足或需要核验图表时临时附带 |
| 主题增量、处理日志和旧版本 | 支线维护与恢复 | 否 | 仅保存在支线本地或归档目录 |

交付边界：支线负责维护上述知识文件，主线只读取并提出 `文献问题_vNNN.md`，不直接改写领域概览、论文目录或主题知识。主线形成的新想法先写入 IDEA 或研究记录；需要文献验证时再交回支线。

主线项目来源更新时，先删除旧版同名文件，再上传本地最新版，避免版本并存。`论文目录.md` 每批同步；主题知识完成一次综合后只同步受影响文件；领域概览仅在跨主题认识实质变化后同步。

用户必须执行的主线同步：将最新版 `知识库/领域概览.md`、`知识库/论文目录.md` 和所有已完成综合的 `知识库/主题知识/*.md` 上传到“科研”主线项目的项目来源。主题知识不需要常驻文献支线项目，但必须常驻主线项目；论文卡和原始 PDF 不在此同步范围内。

## 模式 A：每批五篇新增论文

每个对话默认手动上传五篇论文。附带最新版 `论文目录.md`；五篇明显属于同一主题时，可同时附带该主题最新版知识文件，否则先让 Chat 输出主题归属与主题增量，再在同一对话中补充相关主题文件完成综合。

去重只需把本批论文与最新版论文目录比较：标题、DOI/arXiv ID 或稳定 `paper_id` 任一明确相同，即视为已收录。无需额外检索或考证预印本、会议版、期刊版之间的版本差异；论文没有提供的书目信息记为 `NR`。

> 请遵循项目指令的模式 A 摄取本批五篇论文。先逐篇检查是否与论文目录中的既有论文重复；跳过重复项，再为新增论文分别输出完整论文卡，并输出一份合并后的完整新版论文目录。列出每篇新增论文的主主题、次主题和主题增量；只有已经提供相关主题最新版且本批改变现有知识时，才输出该主题知识或领域概览的完整替换稿。不要生成额外报告。

保存顺序：论文卡 → 论文目录 → 受影响主题知识 → 领域概览。每次确认文件头的 `based_on` 与输入修订号匹配。

## 模式 B：回答主线问题

附带：`文献问题_vNNN.md`、最新版领域概览、相关主题知识/论文卡及必要论文。

> 请遵循项目指令的模式 B，回答 `文献问题_vNNN.md`，输出同编号 `文献地图_vNNN.md`。重点核查反证、最近邻、协议可比性和替代解释；不要生成最终 Idea。

把文献地图交回主线 Chat。若支线同时摄取了新论文，再按模式 A 的保存顺序手动替换知识文件。

## 维护规则

- `领域概览.md` 保持短小，只写跨主题高层认识；建议不超过约 2500 中文字。
- `论文目录.md` 负责覆盖范围和检索，不把论文列表重复写入领域概览。
- 主题知识按任务或机制拆分，只有相关时才附带给 Chat。
- 论文卡使用稳定 `paper_id` 文件名，如 `Xu2025OnlineAction.md`；不要使用容易冲突的 `2025_XXX.md`。
- 交接文件版本递增；知识文件覆盖更新但修订号递增。需要历史备份时由你手动复制到项目外的归档目录。

旧版 `科研文献支线工作流与主线修改方案_v001.md` 仅作历史记录，不添加到 ChatGPT 项目来源，也不参与日常上下文。

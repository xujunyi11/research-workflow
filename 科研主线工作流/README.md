# 科研主线工作流

本工作流旨在让科研思想从一个模糊概念进化为一套完整、可执行的项目文件。

主线只保留三次交接：Chat 交付 Idea，Work 交付实现任务或研究报告，Codex 交付训练证据。文献支线只在可能改变研究判断时介入。

## 一次性配置

1. 在 ChatGPT“科研”项目中，用 `Chat项目指令.md` 全文替换现有项目指令。
2. 从项目来源删除旧版 SOP；把 `Work主线指令.md` 上传为项目来源。不要把它粘贴进项目指令。
3. 选定基线后，把 `Codex主线AGENTS.md` 复制到仓库根目录并改名为 `AGENTS.md`；已有文件时合并而非覆盖。
4. 文献支线使用单独的 ChatGPT 项目，并负责维护主线需要的文献知识来源；主线不直接改写这些文件。
5. 必须把文献支线最新版 `知识库/领域概览.md`、`知识库/论文目录.md` 和所有已完成综合的 `知识库/主题知识/*.md` 上传到“科研”主线项目的项目来源；缺少其中任一类，主线都无法完整使用文献知识库。

### 配置位置对照

| 文件 | 配置位置 | 操作 |
|---|---|---|
| `Chat项目指令.md` | 主线项目的“项目指令” | 全文粘贴并替换旧指令；这是唯一写入项目指令的工作流文件 |
| `Work主线指令.md` | 主线项目的“项目来源” | 作为文件上传，供切换到 Work 时读取；不要粘贴进项目指令 |
| `主线交接模板.md` | 主线与文献支线两个项目的“项目来源” | 主线用其中的文献问题格式，支线用同编号的文献地图格式 |
| `知识库/领域概览.md` | 主线项目的“项目来源” | 上传文献支线最新版 |
| `知识库/论文目录.md` | 主线项目的“项目来源” | 上传文献支线最新版 |
| `知识库/主题知识/*.md` | 主线项目的“项目来源” | 仅上传已经完成综合的最新版主题文件 |
| `Codex主线AGENTS.md` | 基线代码仓库根目录 | 复制并改名为 `AGENTS.md`，不上传到 ChatGPT 项目 |

配置完成后，在主线项目中应看到：项目指令区域只有 `Chat项目指令.md` 的内容；项目来源列表包含 `Work主线指令.md`、`主线交接模板.md` 及三类文献知识文件。进入 Work 后，明确要求它遵循项目来源中的 `Work主线指令.md`。

## 主线项目来源配置

“科研”主线项目应常驻以下来源：

1. `Work主线指令.md`；
2. 文献支线目录中的 `主线交接模板.md`；
3. 文献支线最新版 `知识库/领域概览.md`；
4. 文献支线最新版 `知识库/论文目录.md`；
5. 文献支线中所有已经完成综合、可供主线使用的 `知识库/主题知识/*.md`。

这是部署必做项，不是可选的按需附件。主题知识在文献支线项目中可以只在相关对话按需附带，但其完成综合后的最新版必须同步到主线项目来源；领域概览和论文目录同样必须同步。原始论文和论文卡仍然只按需临时附带。

`主线交接模板.md` 是主线与支线之间的格式契约，不是科研证据。主线需要文献核查时，使用其中“文献问题”部分生成 `文献问题_vNNN.md`；支线使用同一文件中的“文献地图”部分返回同编号 `文献地图_vNNN.md`。模板本身不随每次交接复制或递增，只有工作流格式变化时才替换项目来源中的模板文件。

三类文献知识文件的职责不同：`领域概览.md` 提供跨主题高层结构，`论文目录.md` 负责覆盖范围与论文定位，主题知识负责支撑具体观点、模块、基线和创新空间讨论。主线先查领域概览，再查相关主题知识，最后通过论文目录定位具体论文；证据仍不足时，生成 `文献问题_vNNN.md` 交给支线。

不要把全部原始论文、论文卡、主题增量、处理日志或旧版本知识文件加入主线项目来源。需要核验具体证据时，只在当前对话按需附带对应论文卡、文献地图或原始论文。

文献支线生成新版本后，先删除主线项目来源中的旧版同名文件，再上传本地最新版，避免新旧版本并存。同步频率如下：

- `论文目录.md`：每批论文完成并落盘后同步；
- `主题知识/*.md`：完成一次主题综合后，只同步受影响文件；
- `领域概览.md`：跨主题认识发生实质变化后同步；
- `文献地图_vNNN.md`：作为一次性决策材料交给主线对话，不设为常驻来源。

## 启动与交接

Chat 首次启动：

> 请遵循项目指令，围绕以下模糊方向开始闭环 Idea：……。当前观察/约束：……。先识别关键未知与原因假设，不要提前设计代码。只有文献答案会改变科研判断时才生成精确的文献问题。

主线请求文献核查：

> 当前文献不确定性会影响……。请只生成下一版 `文献问题_vNNN.md`，让它服务一个主要决策，并写明不同答案将如何改变当前 Idea。

文献地图返回主线：

> 请分析 `文献地图_vNNN.md`。先检查证据、反证、最近邻和协议可比性，再说明它如何改变研究闭环；闭环成立时生成下一版 `IDEA_vNNN.md`。

Work 落地：

> 请遵循 `Work主线指令.md` 模式 A，分析 `IDEA_vNNN.md`，核实候选基线并输出下一版 `CODEX_TASK_vNNN.md`。只生成一份任务文件。

Codex 实现：

> 请遵循仓库根目录 `AGENTS.md`，执行 `CODEX_TASK_vNNN.md`；完成后输出 `TRAINING_PACKET_vNNN_r01.md` 及最小必要附件。

Work 分析实验：

> 请遵循 `Work主线指令.md` 模式 B，分析 `TRAINING_PACKET_vNNN_rRR.md`。若值得继续，输出下一版任务；否则输出下一版研究报告。只生成一种交付物。

## 版本规则

| 文件 | 产生者 | 规则 |
|---|---|---|
| `文献问题_vNNN.md` | 主线 Chat | 与同编号文献地图配对 |
| `文献地图_vNNN.md` | 文献 Chat | 精确记录问题及知识文件修订号 |
| `IDEA_vNNN.md` | 主线 Chat | 三位版本号，递增且不覆盖 |
| `CODEX_TASK_vNNN.md` | Work | 独立递增 |
| `TRAINING_PACKET_vTTT_rRR.md` | Codex | 任务版本 + 两位运行号 |
| `RESEARCH_REPORT_vNNN.md` | Work | 独立递增，供 Chat 修正思路 |

大文件、权重、完整日志和论文库留在各自环境；跨窗口只传当前决策需要的最新版文件。
